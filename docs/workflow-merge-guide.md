# 工作流合并指南

本文档用于后续将外置工作流页面逐步合并进 `apps/web` 时参考。当前阶段只保留工作流页面的展位，不直接迁移源码。

## 当前约定

- 工作流是独立一等入口，当前应用通过 `/skills?section=workflows` 进入。
- 当前页面只负责展示工作流位置，不承载工作流内部编辑逻辑。
- 外置工作流通过 `postMessage` 将发布结果回传给当前应用。
- 发布结果应当是完整的 Skill Bundle，后续由当前应用负责创建 `SkillVersion`。

## 现有边界

- 当前应用负责：页面壳、导航、Skill 列表、Skill 详情、版本创建、后续发布承接。
- 外置工作流负责：工作流编辑、转换、bundle 生成、发布触发。
- 两边之间的最小通信协议是 `workflow.publishSkill`。

## 合并时机

建议满足下面条件后再开始内嵌源码：

1. 外置工作流的编辑体验已经稳定。
2. 发布 bundle 的结构已固定。
3. 当前应用的发布承接逻辑已经跑通。
4. 需要统一权限、路由或部署时，再考虑源码合并。

## 合并目标

最终希望把外置工作流从“独立页面”演进为当前项目中的一个可维护模块，同时保留它的产品独立性。

推荐目标结构：

- `apps/web/src/pages/WorkflowPage.tsx`：工作流入口页。
- `apps/web/src/features/workflow/`：工作流的业务组件、状态管理、发布面板。
- `apps/web/src/lib/workflow.ts`：工作流发布协议、bundle 转换适配。

## 推荐合并顺序

### 第 1 步：保留壳，接入远程页面

- 在当前页面继续使用占位展位。
- 通过 `iframe` 或独立窗口嵌入远程工作流。
- 仅实现发布消息接收，不引入业务渲染代码。

### 第 2 步：抽离发布协议

- 把 `postMessage` payload 定义成共享类型。
- 把 Skill bundle 转换结果标准化。
- 把发布确认、错误提示、跳转逻辑留在当前应用。

### 第 3 步：引入工作流 UI

- 将外置页面拆成可复用的前端模块。
- 先迁移静态页面和基础交互，再迁移复杂编辑器。
- 保持发布结果格式不变。

### 第 4 步：再考虑统一部署

- 确认路由、权限、鉴权、静态资源和后端接口都稳定后，再把工作流正式并入主项目。

## 合并时要检查的内容

- 发布 payload 是否仍然包含完整 bundle。
- bundle 中是否始终存在 `SKILL.md`。
- 当前应用是否仍能正确创建新 Skill 或新版本。
- 页面是否仍然保留足够大的编排空间。
- 历史、测评、版本 diff 是否不受影响。

## 当前阶段不要做的事

- 不要直接删除外置工作流。
- 不要急着把复杂编辑器复制进主项目。
- 不要把工作流发布逻辑散落到多个页面。
- 不要让当前应用直接理解工作流内部业务模型。

## 建议的迁移原则

- 先稳定协议，再迁移界面。
- 先保留独立性，再考虑共用组件。
- 先让 SkillHub 接住结果，再让它参与编辑。
- 先最小闭环，再做深度整合。
